### D1 · Modelo de negocio — Tienda completa + suscripción
- **Fecha:** 2026-06-27 · **Fase:** 0.1
- **Decisión:** alimento + accesorios + farmacia, con suscripción/recompra como motor de LTV.
- **Rationale:** compite con los grandes en surtido, pero el diferenciador real es la recurrencia (alimento = consumo predecible = mayor LTV). Se descartó "solo alimento" (mercado chico) y "marketplace multi-tienda" (complejidad excesiva al inicio).

### D2 · Stack técnico — Custom headless Next.js
- **Fecha:** 2026-06-27 · **Fase:** 0.1
- **Decisión:** desarrollo custom headless con Next.js.
- **Rationale:** control total, diseño 100% original, performance de élite. Se descartó Shopify (menos original, fees por venta) y open-source autohospedado puro como única opción.

### D21 · Fase 4 — Stack backend: Medusa.js v2 + principio de producto "e-commerce primero"
- **Fecha:** 2026-07-06 · **Fase:** 4 · **Estado:** LOCKED · **Decidido por Carlos** (elección directa de Medusa, con criterios de priorización explícitos).
- **Principio de producto (rector de toda la arquitectura):** Manada se mantiene como **e-commerce primero**. Perfil de mascota, suscripciones, recomendaciones y anticipación existen **para mejorar la experiencia de compra**, no para convertir el producto en una plataforma compleja de gestión de mascotas. *(Coherente con D17 "e-commerce como piso, perfil como camino destacado" — lo eleva de modelo de UX a principio de arquitectura.)*
- **Criterios de arquitectura (Carlos, en orden de peso):** (1) velocidad de desarrollo y lanzamiento del MVP · (2) simplicidad operativa para un fundador único · (3) bajo costo de mantenimiento · (4) escalar más adelante sin sobreingeniería. **Regla 80–90%:** si una plataforma cubre el 80–90% del MVP y el resto se resuelve con extensiones simples, gana sobre construir desde cero. Estos criterios pesan más que la flexibilidad absoluta.
- **Decisión:** el backend de Manada se construye con **Medusa.js v2** en `apps/backend` (respetando D20). Scaffold en Fase 5; la **versión exacta se fija al scaffold** (mismo criterio que D13 con Next).
- **Validación contra alternativas (resumen):**
  | Criterio | **Medusa v2** | Vendure | Saleor | Custom |
  |---|---|---|---|---|
  | Velocidad MVP | **Alta** — core commerce completo + Admin + mismo lenguaje (TS) | Alta, pero GraphQL-first | Media — Python/Django, segundo runtime | **Baja** — meses reconstruyendo commodity |
  | Fundador único | **1 servicio Node + Postgres + Redis; Admin incluido** | Similar (server+worker) | Stack Python + dashboard aparte | Toda la carga operativa y de bugs |
  | Costo mantenimiento | MIT, sin fees, comunidad TS más grande | MIT core, ecosistema menor | Open core, foco cloud | Máximo |
  | Escala sin sobreingeniería | Modular (módulos/workflows), escala horizontal | Buena | Buena | Depende de diseño propio |
  | Regla 80–90% | **~85% del MVP cubierto; el resto = módulos custom** | Similar pero con capa GraphQL en el front | Menor encaje (lenguaje/stack) | 0% de partida |
- **Por qué Medusa (encaje Manada):** (a) **TypeScript end-to-end** — mismo lenguaje que `apps/web` y que el fundador ya opera (monorepo D20); (b) core commerce (catálogo/carrito/checkout/órdenes/clientes/promos/inventario) **+ Admin operativo incluido** — crítico para un fundador único; (c) **API REST store** que encaja con los providers del frontend diseñados para hidratarse desde backend (D19: `usePet`/`useCart`/`SessionProvider`); (d) framework de extensión de primera clase (módulos con tablas propias, workflows, scheduled jobs, API routes, subscribers) → **el moat (Perfil de Mascota + motor de anticipación) y las suscripciones viven como módulos custom sin fork del core**, y el [recipe oficial de suscripciones](https://docs.medusajs.com/resources/recipes/subscriptions) documenta exactamente ese camino (verificado 2026-07-06); (e) MIT sin fees por venta (coherente con D2, que descartó Shopify por eso); (f) template de deploy en Railway (coherente con la infra prevista en ARCHITECTURE.md).
- **Por qué no las alternativas:** **Vendure** — sólido pero GraphQL-first (capa extra en un frontend pensado REST) y comunidad menor; **Saleor** — Python/Django + GraphQL = segundo lenguaje/runtime para un fundador único, mayor costo operativo; **custom** — viola la regla 80–90%: meses reconstruyendo commodity (carrito, órdenes, promos, admin) antes de tocar el diferenciador.
- **Riesgos aceptados + mitigación:** (1) **suscripciones no nativas** → módulo custom por recipe oficial; además el modelo de Manada es propio (frecuencia por mascota + anticipación), un motor genérico no serviría igual; (2) **sin provider oficial Webpay** (verificado 2026-07-06) → payment provider custom sobre el Payment Module con el SDK oficial de Transbank para Node; existe plugin comunitario de Mercado Pago (minskylab) como segundo medio; la elección fina de proveedores es el siguiente paso de la Fase 4; (3) **Redis requerido en producción** (workflows/eventos) → managed barato (Railway/Upstash); (4) **churn de versiones v2** → versión exacta fijada al scaffold + leer docs de la versión instalada (mismo hábito que `apps/web/AGENTS.md`).
- **Derivados:** `ARCHITECTURE.md §3` actualizado (backend decidido + arquitectura de extensión) y §5 pendientes depurado; **`DATABASE.md` y `API.md` se diseñarán "Medusa-native"** (módulos custom + module links hacia entidades core; API = REST store + rutas custom `/store/*`); orden restante de Fase 4: **proveedores Chile → modelo de datos → contratos API**. El scaffold en `apps/backend` sigue siendo Fase 5 (D20).

### D52 · Storefront — Packshots normalizados server-side (`/api/packshot` + `sharp`): encuadre consistente sin editar assets
- **Fecha:** 2026-07-20 · **Fase:** Fase 5 — MVP (etapa "producto terminado", D38) · **Estado:** LOCKED · **Ejecutada** (`tsc` + `eslint` verdes; **verificado punta a punta contra el dev server real**: `/api/packshot?src=…&w=640` → `200 image/webp` 640×640; `next/image` acepta la URL absoluta de Medusa con `loader` custom y emite el `srcset` `w=64…750` hacia `/api/packshot`; en el pipeline de `sharp`, un asset de **fondo blanco** y uno **transparente** con el mismo producto salen **idénticos**, con el producto ocupando **88.4 %** del cuadro; imagen de un solo color → fallback sin crash).
- **Contexto:** cerraba el hueco marcado por la nota de D51/D18 ("política de imágenes OPEN"). El catálogo real es **heterogéneo**: unos packshots vienen del Admin con **fondo blanco** y mucho aire, otros recortados a **transparente** a mano. Renderizados con `object-contain` + padding por card, el producto se veía **pequeño y disparejo** (captura de Carlos). El fix "por asset" (D51: editar cada PNG a transparente ~90 %) **no escala** a productos futuros y depende de trabajo manual.
- **Decisión:**
  1. **Route handler `GET /api/packshot?src=&w=` (Node, `sharp`):** descarga la imagen del backend Medusa **server-side** (sin CORS), la **aplana sobre blanco** (transparente→blanco; blanco→no-op), **recorta el borde** (`trim`) y la **re-encuadra** en un cuadrado con **margen uniforme** (`marginRatio 0.06` → producto ~88 % del lado), salida **WebP** cacheada (`Cache-Control: immutable`). Así **todo** producto —blanco o transparente— sale con el **mismo encuadre y escala** (DESIGN_SYSTEM §6) **sin editar el asset**. Whitelist de origen (solo backend Medusa / localhost) = no es proxy abierto (SSRF). Degradación: ante fallo de red/decodificación o imagen uniforme → **redirect a la original** (nunca rompe una imagen que hoy carga).
  2. **`ProductImage` graduado a `next/image`** (cumple D36, que dejaba la graduación "para cuando exista foto real"): `fill` + `object-contain` + `loader` custom (`packshotLoader`) que reescribe la URL de Medusa hacia `/api/packshot`. Da `srcset` responsive (varios `w`), lazy, y **cero CLS** (el marco padre `relative` reserva el aspect-ratio). Sin `remotePatterns` (el loader por-instancia lo evita).
  3. **Marcos de media → blanco, sin padding por card:** como los packshots ahora se normalizan **sobre blanco**, los "pozos" de imagen (PDP, quick-buy, cart, recomendación, checkout, food-selector) usan **`bg-white`** para fundirse con el packshot (adiós al borde de caja); se **eliminó el `p-*` por card** (el aire ahora lo da el margen del normalizador, uniforme en todos lados). En la card de PLP el packshot blanco **cubre** el pozo (el gradiente cálido solo asoma para el emoji placeholder).
- **Rationale:** solución **reusable y definitiva** para todo producto que llegue de Medusa (no hay paso manual por asset); consistente entre sí (blanco/transparente indistinguibles tras normalizar); no deforma (aspecto preservado, `fit: contain`); performante (WebP + CDN + `srcset`); sin CLS. Trade-off asumido: el **pozo de media pasa a blanco** (limpio y estándar de e-commerce) en vez de "sobre Arena" cálido — con catálogo **mixto** es el único fondo que se ve consistente para ambos tipos de asset; la calidez de marca vive en el resto de la card, la página y —a futuro— la **fotografía propia** (DESIGN_SYSTEM §6, que mantiene "sobre Arena" para packshots **producidos**, no para los de proveedor).
- **No reabre:** **cumple/extiende** D36 (gradúa `ProductImage` a `next/image`) · **reemplaza** la "Nota derivada (packshots)" de D51 (el fix es frontend, no asset) y cierra la parte de render de la "política de imágenes" de D18 · D29 (suscripción apagada). No toca backend, carrito ni checkout.
- **Nuevo dep:** `sharp` en `apps/web` (ya en `ignoredBuiltDependencies`; binarios prebuilt, funciona en Vercel Node).
- **Archivos:** `lib/media/packshot.ts` (loader + constantes, isomórfico) · `app/api/packshot/route.ts` (normalizador) · `components/commerce/product-image.tsx` (next/image) · consumidores: `product-card`, `quick-buy-card`, `cart-item`, PDP `product-view`, `recommendation-view` (×3), `checkout/page`, `food-selector-dialog`.
- **Derivados:** `COMPONENT_LIBRARY` (ProductImage, ProductCard) · `DESIGN_SYSTEM §6` (packshot) · `CURRENT_STATE.md` (built + frente packshots) · `PROJECT_MASTER §16` · índice de `DECISIONS`.
- **Actualización (impl, mismo día):** el binario nativo de **`sharp`** no cargaba en la función serverless de Vercel → la ruta devolvía **500** y todas las cards mostraban la imagen rota (`alt` como texto). Se probó `serverExternalPackages: ["sharp"]` sin éxito. **Fix definitivo:** se cambió el motor a **`jimp`** (JavaScript **puro**, sin binario nativo → corre en cualquier runtime) y la salida a **JPEG** (el producto va sobre blanco, no requiere alfa). La **arquitectura no cambia** (normalización server-side vía `/api/packshot`, aplana→recorta→re-encuadra ~88 %); verificado contra una imagen real de prod: producto al **88.3 %**, consistente entre assets de fondo blanco y transparente sin importar su tamaño de origen (500×500, etc.). `sharp` se quitó de `apps/web`.
